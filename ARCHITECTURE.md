## рҹҸ—пёҸ PadrГөes de Design Utilizados

### 1. Repository Pattern
- **Objetivo**: Abstrair acesso a dados
- **ImplementaГ§ГЈo**: Interfaces no domain, implementaГ§Гөes na infrastructure

### 2. Dependency Injection
- **Objetivo**: InversГЈo de controle
- **ImplementaГ§ГЈo**: Manual no Main.kt

### 3. Value Object
- **Objetivo**: ValidaГ§ГЈo e imutabilidade
- **ImplementaГ§ГЈo**: Email com validaГ§ГЈo no construtor

### 4. Strategy Pattern
- **Objetivo**: Algoritmos intercambiГЎveis
- **ImplementaГ§ГЈo**: PasswordHasher (pode trocar BCrypt por Argon2)

### 5. Chain of Responsibility
- **Objetivo**: Pipeline de middlewares
- **ImplementaГ§ГЈo**: Javalin before() handlers (RateLimiter вҶ’ AuthMiddleware)

### 6. Factory Pattern
- **Objetivo**: CriaГ§ГЈo de objetos complexos
- **ImplementaГ§ГЈo**: JwtProvider.generateToken(), RefreshTokenProvider.generate()

## рҹҺҜ PrincГӯpios SOLID Aplicados

### Single Responsibility
- вң… `AuthController`: apenas HTTP handling
- вң… `AuthService`: apenas lГіgica de negГіcio
- вң… `JdbcUserRepository`: apenas persistГӘncia

### Open/Closed
- вң… Repositories: abertos para extensГЈo (novas implementaГ§Гөes)
- вң… Fechados para modificaГ§ГЈo (interfaces estГЎveis)

### Liskov Substitution
- вң… Qualquer implementaГ§ГЈo de `UserRepository` pode substituir outra

### Interface Segregation
- вң… Interfaces pequenas e focadas (UserRepository, TodoRepository)

### Dependency Inversion
- вң… Services dependem de interfaces, nГЈo implementaГ§Гөes
- вң… Domain nГЈo depende de Infrastructure
